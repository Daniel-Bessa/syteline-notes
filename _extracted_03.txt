Review how forms, fields, records, and collections 
work in the system  

Mongoose is Infor’s development framework which enables upgradeable extensibility, cloud on premise 
and hybrid deployments, and rapid application development, among other things. The system is built on 
the Infor Mongoose framework. 

System data is stored in tables in its main application database. To access or submit information, you 
generally should use one or more tables in the databases. The system uses fields and other components 
on forms to present collections of data records.  

Forms 

A form is the window through which you interact with information in the system. Each form is devoted to a 
business task or to several related tasks. 

For example, you can use an Attendance form to record and track employee work attendance records. 
Then you can view and manipulate the attendance information by using certain features and 
functionalities in the system.  

Fields 

A field is an area on a form that shows you information that is related to the other fields in a form. It is 
considered a component of a form as are radio boxes, buttons, and other widgets. The information in a 
field is being pulled from a database where it resides. You can also enter and save new data in fields.

Records

How Intelligent Data Objects (IDOs) work with forms  

Intelligent Data Objects (IDOs) work with form metadata and the application database in the system. The 
application database is linked to an objects database and a forms database through configurations. An 
IDO is a business object that contains units of information and logic that are called from the client layer to 
interact with the data in the database. The job of the IDO is to transport collections of data back and forth, 
with any validation or rules needed, between the client and the database. 

An IDO consists of these elements: 

•  A reference to one or more SQL tables  
•  A set of property definitions 
•  A set of zero or more methods 
•  An optional reference to an IDO Extension Class 

In a set of one or more SQL tables each table contains the data for a specified part of the application and 
must include columns (properties) that the system requires to work properly. IDO Properties can be 
bound to columns on tables referenced by the IDO, calculated values, values providing data to server-
side code during query or update, or relationships to child IDOs. 

 The Mongoose framework provides standard functionality for all IDOs: 

•  LoadCollection retrieves a collection of rows from the database.  
•  UpdateCollection takes a set of rows marked for insert, update, or delete, and executes the 

appropriate SQL code against the database.  

•  GetPropertyInfo returns detailed information about the properties supported by the IDO.  
• 

Invoke allows you to execute a custom method. 

How forms use IDOs 

Forms utilize IDOs in many ways. Forms that interact with the application database data define collections 
based on IDOs. There are many types of validators and list sources that are built over IDOs.

Explore the scope of form editing 

The system manages form editing through permissions. Each user account has an edit permissions 
setting which determines what kinds of form changes, if any, the user can make. Form changes are made 
at various levels, called scopes. It is important to understand at what levels a form can be modified. 
Modifications have a very definite order of precedence.  

Let’s look at how permissions are structured in the system by examining the higher-level concept of 
precedence of scope types.    

Vendor version: ScopeType 0 

The vendor version of forms and other global objects is the version that is shipped to the customer. This 
version can only be created and modified by a user with the Vendor Developer edit permissions. In the 
forms database, this shows as ScopeType 0.   

Site Default version: ScopeType 1 

The Site Default version of forms and global objects is an override to any vendor version, and will be 
seen by any user accessing the form, unless there is an applicable group or user version.   

Only users with Site Developer edit permissions can create or change Site Default versions. Site Default 
versions show as ScopeType 1 in the forms database. 

Group version: ScopeType 2 

The Group version of forms and global objects is created for a specific group. Users who have that group 
as their primary permissions group will see that version instead of any site or vendor version. 

Only Site Developers can create or modify Group versions. Group versions show as ScopeType 2 in the 
database, with the column ScopeName specifying the group to which the change applies. 

User version: ScopeType 3 

The User version of forms and global objects is applied to a specific user. Users who have a User 
version of a form will see that instead of any group, site or vendor version.  

Users with edit permissions of Site Developer, Full User, or Basic User can create and edit user versions, 
but users with Full User or Basic User edit permissions can only create and edit user versions for 
themselves. User versions show as ScopeType 3 in the database, with the ScopeName column 
containing the username to which the change applies.

that is being returned from the collection.

properties constitute the method's returned list of values. You can set the parameters that are passed to 
the method. These may include literal values, properties, variables, or components. You also set how the

always the column that provides a value for the component.

Explore the use of validators in the system 

The system provides you with an extensive facility for validating values in components. Validators are 
global objects that can be reused across multiple components and forms. You can associate one or more 
validators with a component.   

Validators can be executed when a user selects from a list or exits the component, or they can be 
deferred until the user attempts to navigate to a different IDO item in the collection or to save the form.  
They support a set of behaviors that allow you to validate a label on a component. For example, you can 
set up a validator that: 

• 

Is related to the component list source in such a way that any value matching an item in the list is 
automatically considered valid.  

•  Validates a value and return output values to the form for functions like displaying associated 

descriptions or setting down-stream component defaults.  

For you to use validators for a component, you must be in Design Mode. Once you open a form and 
select a component you will find Validation under the Data Source section on the Component sheet. 
When you expand Validation, you will see the associated properties.    

To use a validator on a component, you must indicate which validator you want to associate with the 
component in the Validators field. You can indicate the validator properties in the Edit Validators dialog 
box which is accessed when you click the ellipsis in the field.   

You can create, add, edit, or delete a validator in the Edit Validators dialog box. In addition, you can set 
parameters to substitute for the keywords in the error message for the validator. 

Setting validator properties 

Once you have identified the validator for the component, you must set the action that occurs when the 
user interacts with the component. You use the Validate Immediately property to set the timing for the 
execution of the validator. If you set the field to True, the validation occurs immediately when the user 
tabs off the field. If it is set to False, the validation occurs when the user attempts to save the record or 
navigates to another record. 

You also must indicate the error message to display to the user on the Message on Status Bar. You can 
specify a particular string for the error message by selecting True. If you leave it set to the default False, 
it will display a message box that prompts the user to correct or ignore the error. You must specify an 
error message for all validators. 

Using validator messages 

As mentioned before, you can designate substitution keywords for validator messages. These messages 
strings may contain the following substitution keywords: 

•  %V is the value of the current component.  
•  %L is the label of the current component.  
•  %B is an error message passed from a validation stored procedure in a message type parameter.  

Understanding the most common types of validators 

There are different types of validators that you can use. However, the most commonly used ones are:

•  Alphanumeric validators: Alphanumeric validators test whether the value entered is an 
alphabetic character or a numeral. Any text with other types of characters is rejected. 
o  For example, you may use the following attributes to qualify a validation to specify that the 

value in the component should be greater than a specific number: V% > 31.

In Collection validators: In Collection validators can be used to determine whether the

o  For example, you can use %V is not a valid %L to check against a collection list of properties.   

Considerations when using validators  

You should consider the following, when you use validators: 

•  A validator's specification may reference a component, a variable, or a property value.  
• 

If the property value changes after validating the component, then the component is marked as 
not validated. The validator would execute again if the user tabbed through the component or 
saved the form. 

•  For components whose values should be referenced against a table of values, use the collection-

based list sources and validators. 

•  Use the Validate Immediately attribute to eliminate delays from performing all validation 

processing at save time (a behavior that is friendly to run-time users). 

•  Use component classes to package your validators, list sources and other component processing 

for reuse. 

•  Previously validated data may become invalidated because some value on which the validation is 

dependent changes.

Explore the use of strings in the system 

A string is a type of global object that contains a translated literal. When you use string objects rather than 
literal text as form or component captions you ensure consistency and reduce the need for customization 
and translation.  

For example, changing the value of the string named sPartNo from “Part No.” to “Part Number” affects all 
the forms and components utilizing this string. If literal text is used for each caption, each form and each 
component must be updated individually. 

Strings are stored in a strings table in the forms database. The forms database can have multiple 
versions of the strings table. One version of the strings table is current at run time.  

You can create or maintain string objects by using the Edit/Select String dialog box in the system. The 
following actions can be performed: 

•  Create a new string object.  
•  Copy the selected string object.  
•  Edit the value of an existing selected string object.  
•  Delete the selected string object. 

Strings allow for the reuse of captions throughout the entire the system application, including reports.

Add user-extended table fields 

The user extended tables (UET) feature gives the system administrator the ability to extend existing 
application database tables and add custom user fields to forms in the application. Use this feature to 
keep track of information that is not currently in the application database schema. 

If you are using replication, you must click the Regenerate Replication Triggers button 
on the Replication Management form after UETs are changed, added, or deleted. 

Tables are a systematic arrangement of data in records and fields for ready reference. The application 
ships with tables containing predetermined fields. The UET feature allows users to add their own fields to 
these tables. Once a table is extended, you can drop user fields into any application form that uses this 
table. 

If you bind a new component to a UET on a form that uses a custom load method stored 
procedure, an error message displays when you refresh the form. However, the error does 
not prevent you from continuing. 

Adding user-extended table fields 

Use the UET User Fields form to create new user fields, modify existing user fields, or delete existing 
user fields for UETs. Then use the UET Classes form to create new user classes, modify existing user 
classes, or delete existing user classes for UETs. 

  Use the UET Impact Schema form to apply new and changed definitions of User Extended 

Tables to the schemas in all affected databases. The UET Impact Schema form creates physical 
storage for User Extended Table extensions when you make changes. These changes include 
creating new extensions to a database table, or updating a property of a user index or user field. 

The Impact Schema process regenerates triggers for changed tables. To prevent 
blocking, run the process when no other users are logged on the database. 

  After the schema is impacted, restart the application session to refresh the session cache using 

the new schema information. 

  On the UET Impact Schema form, select Commit Form Changes to save the changes you have 

made to the form information using the UET forms.  

  Select Impact Schema to change the database schema to contain the columns and indexes 

corresponding to the user fields you have defined in the UET forms. Corresponding views over 
multi-site tables are also updated. Then, click Process.

Understanding event handlers 

There are two types of events and event handlers in the system: form events and application events. 
They are similar, but the key distinguishing factor, though, is whether they are form-only in scope or 
application-wide. This lesson focuses on form events.   

Event handlers define a unit of work to be performed in response to an event. Event handlers are defined 
for specific forms. They are not shared across forms.  For example, the types of event handlers that can 
be used on a form are: 

•  Goto URL 
•  Run Form as Linked Child  

Event handler sequence 

Event handlers are identified by the name of the event that they are handling and by a sequence number. 
You can define multiple event handlers for any event. These event handlers execute in sequence number 
order.  

By default, if any event handler reports a failure, the remaining event handlers in the sequence are 
skipped. You can override this default by setting the Ignore Failure parameter. If you set this attribute, 
the event handler will always be treated as successful. 

Built-in responses 

When you define additional event handlers for standard events that have built-in responses, the built-in 
responses always execute last. Consequently, you can cancel the built-in responses by reporting failure 
from a custom event handler for the same event. 

There are many different response types to use as event handlers. The main categories are: 

•  Collection processing 
•  Execute an executable 
•  Form navigation  
•  Generate event 
•  Goto form 
• 
Inline script 
•  Method call 
•  Run background task 
•  Run script 
•  Set values

Why you need FormSync 

As a Systems Administrator, you have been assigned the task of making some significant changes to a 
form. As best practice dictates, you would not start to make changes to the form in question in your 
production environment. Hence, you would log into your test environment and commence the process of 
making the relevant changes and carry out the testing of those changes. 

The next step in the process would be to move all the changes you made into your production environment 
without having to duplicate your efforts. This is where FormSync becomes an invaluable tool in that with 
the click of a single button, you are able to duplicate all your changes from a source system to a target 
system. 

Overview 

In Design Mode, WinStudio serves as a development tool for customizing forms in Mongoose-based 
applications. FormSync is an aid to preserving your customizations when you upgrade to a newer version 
of Mongoose or your Mongoose-based application; when you apply a service pack or a single issue fix; or 
when you distribute and apply your own forms, form modifications and other global objects in Mongoose. 
The process of handling these changes during an update or upgrade is referred to as synchronization. 

Note: Mongoose also incorporates much of the functionality described in this guide within the Form Sync 
form. But the reader should be aware that not everything described in this guide applies equally to that 
form. For more information about the differences, see the online help for that form. 

FormSync also provides a set of development and administrative utilities with which you can: 

•  Gather information about customizations. 
•  Copy customizations from one database to another. 
•  Delete objects. 
•  Compare objects. 
•  Generate SQL scripts to apply customizations to a remote database. 
•  Generate "pseudo code" for forms and objects to use later in evaluating and updating 

customizations.  

The Nature of Customizations 

FormSync looks for differences between a source version and a target version. Any type of change that 
has been made to an attribute value registers as a customization in synchronization, whether or not the 
change affects functionality. 

For example, if a customer developer adds a comment to a script, FormSync considers the script to be 
customized, even though the functionality of the script has not changed. 

A set of specifications for a complex object, such as a component class or a script, is treated as a unit in 
synchronization. Mongoose does not attempt to selectively synchronize individual specifications for the 
object. 

The relationship between Global Objects and Forms during customization 

Global objects are listed as attributes of forms and components in the Merge Options dialog box. The 
values of these attributes are named references to global objects. A customization of a form component 
with regards to global objects consists only of selection of a different named object in the component 
specifications. 

118   Lesson 11: Form Sync 

© 2018 Infor Education. All rights reserved. No part of this Training Workbook may be reproduced or transmitted in any form, without written permission. 

 
 
 
 
For example, if a component uses the right-click menu named StdDefault and a customer developer 
changes the value for a component to StdDetailsAddFind, FormSync processes the change as a 
customization of the component, not of the form that uses it. Customizations of global objects themselves 
are processed separately, as specified in the Synchronization Options dialog box. If you change the 
attributes of the menu StdDefault, the menu is processed as a customized version of the global object, 
but form components that use the customized menu are not processed as customized. 

The Basic Synchronization Process 

For any event handler, you can define messages to be displayed when the event handler fails or During 
synchronization, FormSync compares new versions of forms and objects (in the Source database) with 
existing versions of those forms and objects (in the Target database). FormSync also compares the Source 
versions with any existing customizations in the Target database. Depending on your synchronization 
settings, FormSync then either keeps the customizations, merging them with new data from the Source 
database, or removes and replaces customizations with the new data. 

Finally, FormSync replaces the existing vendor versions in the Target database with the new vendor 
versions from the Source database. For more information, see Chapter 2, “Schematic Overview of 
Synchronization." 

Conflicts Between Customizations and New Versions 

FormSync does not attempt to resolve conflicts between the size and position of customized components 
and revised settings for size and position in a new base version. Where you might have moved, resized, or 
added components on a form, those components can display overlapped with a base version of a 
component in a merged form. 

Automatic Conversions of Scripts  

One of the major changes for FormSync Version 8.0 (which has been carried forward to subsequent 
verions) was the conversion of VBA code to .NET code. During synchronization, FormSync evaluates 
both form and global scripts to determine whether they are VBA code, which was used in earlier versions 
of applications (such as SyteLine) built on Mongoose. If the code is determined to be VBA, then 
FormSync attempts to convert the code automatically to VB.NET. After converting the code, FormSync 
prompts you to keep, remove, or edit the new version of the script. 

If you choose to: 

•  Keep the converted code, FormSync commits the automatically converted code to the Target 

database. 

•  Remove the converted code, FormSync removes the customized script and replaces it with the 

script from the new (Source) version. 

•  Edit the converted code, FormSync opens the file comparison utility with the new Source version 

of the script in one pane and the customized code that FormSync converted to VB.NET in 
another pane. This allows you to compare versions, evaluate the new converted version, and 
make changes in the customized version if necessary, before FormSync commits the changes to 
the Target version. 

Note: Whether you keep the converted code as is or edit and then save it, there are no 
guarantees that the resulting code is fully functional. Plan to test the script before placing it into a 
production environment.

Planning Customization and Synchronization 

As a development project, customizing WinStudio forms requires planning, documentation, and 
application of development policies. FormSync cannot anticipate the consequences of all customizations 
or guarantee the functionality of both customizations and base-level functionality after an update is applied. 
FormSync is most effective in maintaining functionality when it is used in conjunction with a development 
plan, documentation supporting the plan, and testing. 

FormSync Administration Utilities  

The functionality of FormSync in the update process is different from its functionality as a development 
tool for administering forms databases. 

In an update, synchronization handles vendor versions of forms in both the revised and the unrevised 
versions of a Mongoose-based application. FormSync compares any customized version of a form with 
the two vendor versions of it. FormSync then allows you to update the customized version with 
specifications from the new vendor version, while selectively keeping customizations. Alternatively, 
synchronization can remove customized versions entirely or keep them intact. In all cases, FormSync 
replaces old vendor versions with new vendor versions. 

FormSync’s utilities for copying, deleting, and scripting objects do not affect vendor versions of forms or 
global objects. These utilities access only customized versions. These utilities are useful primarily to those

Appendix B: Creating a new form 

Copy a form 

Before anything is explained in too much detail, it is important to understand a common-sense guideline 
in creating new forms. 

It is very rare that a form will need to be created entirely from scratch. In most cases, a similar, existing 
form can be copied and modified. 

To copy a form, begin by opening a form in Design mode. Then, select System > Form > Definition > 
Copy from the main menu. The Copy Form Definition dialog box will display. 

•  Next, select the form that you wish to copy from the Source Form drop-down menu. Then, type 

<a name> in the Target Form field. 

•  Check the Copy Template Also check box to copy the template and click OK. 

You can now search for, open, and modify the form to fit your needs. 

Create a new form 

The first thing that you should understand before creating a new form is the difference between bound 
and unbound forms. 

A bound form is one that is used to retrieve and manipulate one or more collections. (A collection is the 
result set returned by an IDO.) The relationship between the form and an IDO is determined by specifying 
the collection to which the form is bound. In other words, a form is associated with the result set that is 
returned by a particular IDO. 

An unbound form is one that is not tied to a particular IDO. It may be used to simply gather data to pass 
to a method or procedure, such as a report generating form. 

To create a bound form, make sure you have Site Developer permission. Then, go into Design Mode. 

•  Next, select System > Form > Definition > New. The New Form Wizard screen will display. 
•  Select Create a New Form, click Next, and type the <form name> in the field.  
•  Add attributes in the Add Attributes section; this includes selecting a data source from the Data 

Source drop-down list. 

•  Then, click Next. 
•  Select properties for the form and then click Next.  
•  Finally, select the order of the attributes on the Properties screen and then click Next > Finish.
