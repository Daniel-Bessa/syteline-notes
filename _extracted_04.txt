About this workbook 

Welcome to this Infor Education course! We hope you will find this learning experience enjoyable and 
instructive. This Training Workbook is designed to support the following forms of learning: 

•  Classroom instructor-led training 
•  Virtual instructor-led training 
•  Self-directed learning 

This Training Workbook is not intended for use as a product user guide. 

Workbook design 

This Training Workbook contains both conceptual information to introduce topics, and step-by-step 
procedural instructions for practical application of those concepts. Conceptual information is included 
within each lesson, while the corresponding step-by-step procedural instructions are included in the 
Exercises and demos section at the back of this Training Workbook. 

Symbols and notes are provided throughout this Training Workbook for ease of reference. Refer to the 
Symbols used in this workbook section below to familiarize yourself with these symbols.  

Hyperlinks have also been included to assist with navigation. For example, if you are viewing an 
electronic copy of this Training Workbook, use the links provided with each demo or exercise to quickly 
navigate between the conceptual lesson content and the procedural demo or exercise instructions.

Course training environment

Education Training Environment needed to complete the exercises and demos. Your instructor will also 
assign user account logins and passwords from Appendix A of this Training Workbook. 

If you are taking this course as self-directed learning (SDL), refer to the Lab on Demand page in the SDL 
course for instructions about accessing and launching the corresponding Infor Education Training 
Environment, including user account information required for access. 

The exercises and demos in this course build upon each other as they prepare the system for 
subsequent exercises and demos. You must complete all of the exercises and demos in the 
order in which they occur in the Training Workbook. This ensures you will achieve the 
expected results and will be best positioned to successfully complete the course.

User interface 

Standard user interface elements include the navigation pane menus and the main toolbar. 

Navigation pane menus and main toolbar 

Navigation pane menus 

You can browse all available folders through the navigation pane menus and submenus. Click the down 
arrow for each of the navigation pane menus, this allows you to explore the submenus contained 
beneath. 

Navigation pane menus and submenus 

For information on how to pin submenus to the navigation pane, please refer to the Creating Form 
Personalizations workbook or ILT. For example, if you know that you will be using the Customer Service 
submenus frequently, you can display the menus at the top level, by pinning them to the navigation pane. 

Main toolbar 

The main toolbar contains icons that allow you to quickly complete commonly used tasks, such as saving 
data or closing a form. You can position your mouse pointer over an icon on the toolbar to reveal its 
description and functionality.  

To view a list of all toolbar icons and for steps on how to add/remove icons from the default toolbar, See 
Appendix B. Note: Availability to make hiding buttons visible is not available for Web users. 

Keyboard shortcuts 

Some navigation pane submenus display available keyboard shortcuts that can be used to save multiple 
steps when completing certain tasks.

Mongoose is the Infor development framework that was used to build the system. The entire user

Mongoose can be used to develop new relational database applications as well as extend the 
functionality of applications that were built on the framework; CloudSuite being one of those applications. 
You will use the Infor Mongoose development framework during this course to add new forms and 
modify/extend an existing system form.

Running forms in CloudSuite 

Any task you work on in the system is completed using some type of form. To “run” a form in the system 
means to display the data in the form that is pulled from the database. Because much of the Mongoose 
development is done using forms, getting comfortable with the productivity features provided in the 
application is helpful to support effective development. 

Form modes 

Forms can be placed in various modes within the system including: 

•  Filter In Place mode 
•  Run mode 
•  Design mode 
•  Form-only mode 

Filter In Place mode 

The Filter In Place mode is used to filter criteria on the form to return a collection of related data from the 
database. This mode is a common default set for forms when they are first opened. You trigger the Filter 
In Place command to run the form when it is in Filter In Place or Design mode.  

Run mode 

The Run mode is used to work with the filtered collection of records. This mode results when you trigger 
the Filter In Place command, while in Filter In Place (FIP) mode, to run the filter. You can filter to retrieve 
a collection of one or more records and the resulting forms are ready to be viewed and edited. 

Design mode 

The Design mode is used to modify form component properties and overall page layout of all 
components. You cannot use the form in this mode. You must change to Run mode to view and edit the 
data that is displayed in a form. You will use this mode extensively throughout this course to extend a 
system form and modify forms you create using the wizards. 

Form-only mode 

The form-only mode is used when a form is presented with its own navigation features in a web page. 
Any form can be presented as a stand-alone web page, without the normal frame and menus, using a 
URL parameter. Forms run in this fashion are said to be run in form-only mode. This technique is used for 
web portals, smart phone displays, etc. and we will use this technique for our expense reports portal in 
Lesson 7.

The relationship between form metadata, IDOs, and 
the application database 

Development in Mongoose/CloudSuite involves working with: 

•  Tables that contain data. 
• 
•  Forms that use IDOs for their collections. 

IDOs that provide access to tables as well as other functionality. 

All development for creating and modifying tables, IDOs, and forms can be done in WinStudio Design 
mode. The system provides core forms that you use to edit the schema (tables and columns), IDO 
definitions, and create and develop forms. 

The goal for using the Mongoose framework is to let you do as much development as possible without 
having to write code. 

IDOs 

An IDO is a business object that contains units of information and logic that are called from the client 
layer to interact with the data in the database. The job of the IDO is to transport collections of data back 
and forth, including any required validation or rules, between the client and the database.  

An IDO is a component that implements a set of methods and properties used by clients to perform tasks 
such as loading and updating collections or retrieving information about data. Related IDOs are typically 
grouped into an IDO project and Mongoose/CloudSuite provides the IDO Projects form to create and 
manage IDO projects. Access this form by navigating to Master Explorer > Modules > System > IDO 
Editing > IDO Projects. 

An IDO consists of the following elements: 

•  A reference to one or more SQL tables  
•  A set of property definitions 
•  A set of zero or more methods 
•  A reference to an IDO extension class (optional) 

For each IDO, the Mongoose framework provides the ability to: 

•  Load and update collections and invoke IDO-specific methods. 
•  Query metadata about the IDO and its properties. 

One or more properties are designated to be the IDO’s identifier or key. For example, a customer IDO 
might have the following properties: 

•  Customer ID 
•  Customer name 
•  Customer address 

IDO forms serve as a development environment for IDOs. IDO definitions are stored as metadata in the 
Objects DB and the metadata can be edited using the IDO forms. You maintain IDO definitions using the 
IDOs form, which can be accessed by navigating to the Master Explorer > Modules > System > IDO 
Editing > IDOs.

SQL tables 

All IDOs include a reference to one or more SQL tables in the application database. Each table contains 
the data for a specified part of the application, with columns on the tables bound by IDO properties as 
needed. Although there are many aspects of IDOs that do not correspond directly to tables in a SQL 
Server® relational database, SQL Server tables provide the foundation on which any IDO is built. The 
following are requirements for SQL tables and IDOs: 

•  All tables used by IDOs must have certain required columns and triggers.  
•  Each IDO must have at least one base table.  
•  Secondary tables are used to include read-only information from associated tables in an IDO.  
•  For most IDOs, primary key properties used by the base table in the application database are 

detected automatically.  
o  Primary key properties for IDOs based on a SQL view must be specified manually.  
o  To identify the primary key properties and their sort order, set the IDO's advanced attributes.  
You create, modify, or delete SQL database tables using the Sql Columns, Sql Tables Constraint, and Sql 
Tables forms, which can be accessed by navigating to Master Explorer > Modules > System > Schema 
Editing. 

Do not delete or modify base application tables or columns. Doing so can cause system

Unbound properties 

Unbound properties contain values that are not stored in a database table and consequently are not

Subcollection properties 

Subcollections are the principal mechanisms for defining hierarchical or parent-child data relationships. 
Use subcollections to implement one-to-many relationships between IDOs. A subcollection property is a 
property that specifies a child IDO that is filtered from a parent IDO. A subcollection is the child IDO 
whose returned collection is associated with, and dependent on, the objects returned in the primary 
collection belonging to the parent IDO.  

Example use of a subcollection 

The typical implementation of order lines in a business application is a good example of a subcollection. 
Each order returns a collection of order lines. The system would define the order lines as a subcollection 
of the order IDO. Note that order lines are dependent on orders; that is, order lines cannot exist 
independently of the order’s parent. However, this is not a requirement of subcollections in general. For 
instance, you may define a collection of customers as a subcollection of the collection of account 
managers. Each account manager has a set of customers that he or she services, but customers can 
exist totally independently of their account managers. It should be possible, for instance, to move a set of 
customers to a new account manager and delete their old account manager from the system. 

Subcollections can also be used to implement recursive data structures. A good example of this is a 
typical implementation of a product structure or bill of materials. A product structure record typically 
includes a reference to a set of child entities that are themselves product structure records. In this case, 
you could define a product structure’s IDO with a subcollection of product structures. 

To define a subcollection, you must first define the collection class that characterizes the child IDO. Then 
you must establish the relationship between the new child IDO and the parent IDO by creating a 
subcollection property on the parent. In the case of orders and order lines, you would define an orders 
collection and an order lines collection. Then you would define the relationship between the two by 
defining a subcollection property on the orders IDO. 

Methods 

IDOs can contain a set of zero or more methods. A method is based on a stored procedure in the 
application database or an extension class method (e.g., .Net code). You add methods to IDOs using the 
IDOs form which can be accessed by navigating to Master Explorer > Modules > System > IDO Editing 
> IDOs. IDOs have two kinds of methods—standard and custom load. 

Standard methods  

Standard methods do not return a result set. They take zero or more parameters and return a result. 

Custom load methods 

Custom load methods return a result set and can be used for lists and collections in forms.

uses it. This provides a single point of maintenance for the core data types in your application.

•  A new form is created and saved to the Forms database. 
•  The IDO metadata cache is emptied. 
•  The Sql Tables form is launched and filtered for the new table. 
•  The IDOs form is launched and filtered for the new IDO. 
•  The new form is launched in Run mode.

Working with property class extensions 

Adding form component drill-down functionality 

Many form components are intended to maintain values that refer to items that are maintained elsewhere 
in the system. An order-entry form, for instance, may contain a Customer ID component that is used to 
reference the customer placing the order. However, the actual customer is typically maintained in the 
Customer Collection. 

WinStudio provides a built-in facility that makes it easy for the user to navigate from the component that 
references an item to forms in which the item can be maintained or queried.  

Component navigation behavior 

Shortcut menus for form components support three types of navigation type behaviors that are described

Using the AUTONUMBER() keyword as a default value 

As you can see in the screenshot above, the Default Value field is set to AUTONUMBER(). This is an 
example of a substitution keyword that is available for use in the Mongoose framework.  

Default value fields 

One of the attributes you can define on a property class, and override all the way out to components on a 
form is the default value. When a New operation occurs on a form, this value will be placed into the 
components, and if an UpdateCollection (i.e., a Save operation) occurs through the IDO and no default 
value has been supplied for a new row, the default value for the IDO property will be filled in prior to 
executing the SQL Insert statement to the database.  

Mongoose substitution keywords 
In WinStudio, a Mongoose substitution keyword is a type of keyword that returns certain system or form 
values. For example, the substitution keyword USERNAME() returns the name of the current user. In 
previous lesson exercises, you set the Default Value field to AUTONUMBER() for the property base 
classes so that the form fields that inherited this value defaulted to the next available number, starting 
with the number “1.” There is an interpreter built in to the WinStudio runtime that is also utilized in some 
cases in the IDORuntime. When WinStudio interprets a substitution keyword, it replaces the keyword and 
its parameters with a value. 

Leveraging the keywords interpreted by Mongoose helps reduce the amount of code you have to write. In 
this scenario, the keyword AUTONUMBER() will be used for a new base property class that you will 
create called MGCExpenseReportIDBase.  

Using the CURDATE() keyword as a default value 

Another substitution keyword that you will use in this lesson is CURDATE() which evaluates to the current 
date. Therefore, when CURDATE() is entered into the Default Value field for the From Date field on the 
Expense Report, the current date that is found in the system will automatically display in the field when 
the form is run. Note: The current date displayed is corrected for the time zone of the database server.

Using subcollection grid components 

A grid component is a WinStudio form component that allows data to be displayed and maintained in a 
tabular (row-column) form. The data in a grid is taken from an IDO collection, which must be bound to the 
grid. The grid component itself is a container for one or more grid columns. Each column displays data 
associated with a single property from the IDO collection.  

A grid can be bound to a primary collection, a secondary collection, or a subcollection. A collection bound 
to a grid is displayed and maintained in the grid. A subcollection is filtered to show results that are 
associated with a primary collection, where there is a one-to-many relationship between an item in the 
parent (or top-level) collection and one or more items in the child (or sub) collection.

Using the WinStudio event model 

The WinStudio form event system lets you define logic with no or minimal code. At its core, the form event 
system uses what are called events and event handlers. WinStudio defines a fairly comprehensive set of 
standard events, most of which have responses built in. There are two kinds of events that can be 
generated in WinStudio: 

•  First, standard events are generated automatically as the user interacts with the WinStudio client.  
•  Second, custom events can be created by the WinStudio developer and generated based on the 

conditions specified by the developer. 

By themselves, events do not perform work. They simply notify WinStudio that something has happened. 
The actual work is performed by event handlers. Event handlers define units of work (i.e., responses) that 
are associated with a specific event type. The work is executed whenever an event of that type is 
generated. WinStudio provides built-in responses for some standard events. It also provides a facility for 
creating custom event handlers. You can create custom event handlers for all standard and custom 
events. 

Some reasons that you might want to create event handlers include the following: 

•  To handle the very simple and very common case of performing some work based on an action 
by the user. For instance, you might want to launch a form or call a method when a button is 
pushed. In this case, you would create a custom event associated with the button and create an 
event handler for this event that performs the required work. There are a variety of response 
types that can be used, including the execution of methods and scripts.  

•  To override the standard WinStudio behavior. In these cases, you may want to add some 

processing and/or cancel the default processing that happens in response to some standard 
event. These adjustments can be easily accomplished by creating event handlers for the 
standard event. You can generate these standard events yourself in various ways, or add your 
own handlers to them so that as the user interacts with your forms your logic is introduced and 
can even cancel the default behaviors. You can also define your own events, creating event 
handlers for them, as you will do next with the date fields on the Expense Reports form. 
Introduce whatever logic you need to add to your user interface, to an arbitrary level of 
complexity.  

• 

The form event system is one of the main tools you have for minimizing the amount of code you have to 
write. Event handlers have response types, which are a wide set of actions you can use, like navigating to 
other forms or web pages, querying data on various collections on your form, manipulating binary values, 
and executing some code. 

There is a second event system available for use within Mongoose called the Application 
Event System. The Application Event System (AES) has the same goal for minimizing the 
code you need to write to define the behaviors of your application, but the events and event 
handlers in AES execute on the app server, and have a different set of actions, as well as a 
different set of standard events defined by the framework. This course does not cover use 
of AES. 

There are a couple special contingencies that come in to play when a form includes multiple collections. 
By default, event handlers execute on all collections, regardless of whether they are active or not. This 
can have undesirable results.

You can define events to be generated when the user interacts with a component. Most components

Using forms as stand-alone web pages 

You have created forms in previous lessons that are intended for use within the application frame (i.e., 
menus, toolbars, etc.) and now you will create a form that is intended for display and use in a Web 
browser. A nice feature of the Mongoose framework is that all elements you develop can be reused or 
repurposed. You created IDOs, tables, and a form for expense reports in the previous lesson and now 
you will build a new portal version of the Expense Reports form based on the existing underlying IDOs 
and tables that the Expense Reports form uses. 

To launch a form as a stand-alone web page, you will use the page-formonly parameter in the URL. 
Users will access this portal with the link formatted accordingly. The users must still sign in but will only 
access the form provided and no other CloudSuite/Mongoose features would be accessible from this type 
of access. This Expense Report Portal form will allow users to add and modify their own expense reports. 

Form-only mode 

You will build the portal form to run as a stand-alone web page. Virtually all Mongoose forms can be 
viewed and used in a Web browser. They can also be rendered as stand-alone web pages, without the 
Explorer or menus showing. This is called form-only mode and is controlled by the link used to access the 
Mongoose application in the Web browser. Presenting a form this way allows you to design the UI for 
users that have no training using the standard menus and toolbars, making it function however you want.  

Because you will not be using the built-in CloudSuite/Mongoose menus and other navigational elements, 
you will have to build all navigation into the form. The Mongoose event system includes actions you can 
use, including changing form pages and launching other forms, which will replace the current form in the 
browser until that form closes. You will use these features in this lesson’s exercises. 

WinStudio FormPage component 

You will use the WinStudio FormPage component to incorporate multiple screens, or pages, within a 
single form. Refer to the “Design mode controls” section in Lesson 3 for a brief overview about the 
FormPage component.

Adding a permanent filter 

Normally, when the user logs in to the portal form that you are creating, you want the user to see only 
those expense report items that the user created and by a specific date. In a future exercise, you will 
create a QueryPage on the form to search the user’s own expense reports and the QueryPage will only 
display the expense reports that were created by the user.  

This functionality is accomplished using what is called a Permanent Filter Expression, which is a form-
level property that you can use to automatically and permanently filter a collection when the form is 
opened. You will set this property using a form variable to specify the criteria the form is to use to filter the 
collection when the form is opened by typing MGCUsername=‘USERNAME()’ into the Permanent Filter 
Expression property field on the SLEmployees primary collection. You will also set this on a hidden grid 
that is used just for this purpose in the background. 

Taking the glue off the form component 

A very useful set of properties for many form components is the properties that allow you to “glue” edges 
of those components so that they adjust with their container. These properties are located under the 
Layout node in the Form Component properties. 

There are four glue properties, one for each side of the form, and they allow you to control how 
components move or resize when their containers are resized. You can place components within a form 
container, such as a notebook tab or a form page, or directly within the form itself.  

When users resize the container: 

•  Glued components move or resize proportionately, depending on how these glue properties are 

set.  

•  Unglued components retain their original sizes and positions, regardless of how their containers 

are resized, sometimes resulting in the addition of scroll bars if the container is sized down too 
much. 

These glue properties are initially set by the New Form Wizard at the time the form is created. In the 
following exercise, you will select the Grid Only form type option and the wizard will automatically set the 
Bottom and Right glue properties to a True value. However, you will change the default settings so that all 
sides of the grid component are set to False.

Adding scripts to a form  

The main benefit to developing applications using the Mongoose framework is the fact that minimal 
coding is required to develop; however, there will be times when you need to add code. You can add 
code into the user interface, using what Mongoose calls form scripting.  

Types of scripts 

Mongoose makes available three basic types of scripts you can use in your forms: 

•  Form script 
•  Global script 
• 
Inline script  

Form scripts 

A form script is a script that can be used only within the confines of a single form. It is not available to 
other forms or other applications. Up to this point, you have completed all necessary tasks without having 
to compose or use a single line of code directly; however, the need to use code is not always avoidable. 
In the next exercise, you will very briefly work in the WinStudio Script Editor to add a few lines of code 
using a form script. This will be a simple bit of code used to make the sign out hyperlink work correctly on 
the Expense Report Portal form.  

Global scripts  

A global script can be used anywhere in the application, by any form. Global scripts are “global” in the 
sense that they can be used and reused with any form in an application—or any part of a single form.  

Scripting language 

When it comes to choosing a language, there are several ways and places to specify the language you 
want to use:  

•  On a system-wide level, you can use the Process Defaults form to make settings that affect all 

developers in your system.  

•  For form scripts, you can set the script language using the form-level property, Script Language.  
•  For global scripts, you can set the script language at the time you create the script, assuming the 

Allow script language selection process default has been set to one.  

Inline scripts  

Inline scripts are limited to 1500 characters, and thus are of limited use, especially for our purposes in this 
course.  

Script editors  

Whatever language you decide to use, you are going to use a script editor to write and compile your 
code. Mongoose has a built-in script editor and you will use it in the next exercise. The built-in script 
editor is a basic, stripped-down tool that meets our needs for this course. However, for serious code 
development, you will likely want something more powerful.  

Mongoose supports the following script editor, all based on Microsoft’s Visual Studio®: 

•  WinStudio

Using the FormPage component 

Now that you have started building out the Expense Report portal form, you will add the FormPage 
components that will be used to selectively display different content pages—HomePage, AddPage, and 
QueryPage.  

Containers  

Containers are components that are used to hold and display other components, and the FormPage is 
one type of container. Typically, they are of no use by themselves, they must “contain” other components 
that actually do something.  

CloudSuite/Mongoose has three types of components that are used as containers: 

•  Grid component type: 

o  The grid component is used only as a container for Grid Column components.  

•  Notebook component type: 

o  This component type is used as a container for Notebook Tab components.  
o  This component is typically used when you have too many related components to fit on a 

single form, and you don’t want to create multiple forms.  

•  FormPage component type: 

o  This component type is, in many ways, the most flexible of the container component types.  
o  This container can contain a wide variety of other components, mostly the same as a regular 

form.  

o  By adding multiple pages, you can, in effect, present what appears to be multiple forms within 

a single form. 

•  FlexLayout component type: 

o  A FlexLayout component is a container component which is used to flexibly house other 

components.  

o  Each FlexLayout component can be divided into different horizontal or vertical regions.  
o  each region can contain multiple child components. 
o  The overall positioning of those child components is specified by the FlexLayout region 

attributes, and their order is specified by the Sequence property. 

In Runtime mode these form pages are set up to display and enable only the “current” page. This means 
that you must set up components within each form page that allow you to navigate between all pages in 
the form. One of the more flexible features of the FormPage component is the ability to add other 
components so that they can be shared on all (or multiple) pages, while other components can be 
accessed only on one specific page. 

FormPage components are typically used to create wizard-type forms. You have already experienced this 
type of form when you used the New IDO Wizard in a previous lesson exercise. 

Container vs contained relationship 

The container/contained relationship is usually established simply by adding the contained components 
on the container. This works for grid/grid column relationships and notebook/notebook tab relationships, 
but may not possibly be so obvious on a form page. You can create components to be contained before 
the container, and it is possible to create or modify the relationship after the fact. You do this by using the 
Container property on the component’s property sheet.

Adding the form page navigation 

We discussed the WinStudio event model in a previous lesson. Now you will work again with events and 
event handlers for the HomePage buttons that will be used to navigate to the AddPage and QueryPage.  

In this system, predefined events trigger predetermined responses from Mongoose. These responses can 
be the result of user interactions with the form or generated programmatically as the result of activity 
taking place in Mongoose itself.  

At its core, the form event system uses what are called events and event handlers to perform these 
behaviors. As you know, WinStudio comes with a fairly comprehensive set of standard events and event 
handlers that are built-in. These standard events and event handlers include a wide range of behaviors, 
so that you do not have to constantly program and reprogram them yourself. But, for those cases where 
WinStudio does not have an event or event handler that does what you need, Mongoose also provides 
the capability for you to define your own custom events and event handlers.  

The form event system 

The primary reason the form event system was incorporated into the Mongoose framework was to 
minimize the need for form developers to write a lot of code. At the same time, the framework provides 
developers with a comprehensive set of standard behaviors that can be set to execute automatically—in 
addition to the ability to further broaden the set of behaviors that can be used in this way.  

Events 

As we discussed previously, an event is merely an action, or a condition that WinStudio is programmed to 
recognize as requiring a response. It might refer to a user clicking on a button, the system performing a 
calculation, or the mere movement of time. An event, as defined in the system then, is an action or a 
condition that WinStudio recognizes and which triggers a response from Mongoose. The response 
mechanism is called an “event handler.” For an event to be meaningful, it must have at least one 
associated event handler that tells WinStudio how to respond.  

Event handlers  

That means that an event handler is essentially a set of instructions that Mongoose uses to respond to a 
particular event. Response types are used to determine exactly how the system should respond to the 
event. Many of these response types prompt the developer for additional specifications or information for 
the system to take into account when responding. This means that developers can specify additional 
actions or alternate actions for the system to take. Because any built-in actions for a standard event 
handler are performed last, this means that you can specify one or more actions that are performed first 
and that can even be used to circumvent the built-in actions altogether.  

Event handlers also provide options for the developer to communicate information to the user that might 
be helpful in knowing that the action succeeded or the reason why an action has failed. These messages 
are predefined and “attached”, to the event handler. They can contain strings, either literal or translatable, 
and keywords, which are sort of like string “variables”—i.e., values that the system picks up at runtime 
and substitutes for the keywords when presenting the message. By default, messages display in popup 
message boxes, but you can, if you wish, set them to display in the status bar instead.  

Multiple collections in forms 

There are a couple special contingencies that come into play when a form includes multiple collections. 
By default, event handlers execute on all collections, regardless of whether they are active or not. This 

v10 Extending the Application with Mongoose Training Workbook (v10.18)  169 

© 2018 Infor Education. All rights reserved. No part of this Training Workbook may be reproduced or transmitted in any form, without written permission. 

 
 
 
 
 
 
 
can have undesirable results. So, Mongoose makes it possible to create event handlers in such a way 
that they operate only with a particular collection. This is done using the Only When Current Collection 
parameter when defining the event handler. When that parameter is specified, then the handler responds 
only when the specified collection is the currently active collection.  

Event handlers—response types 

Now let’s talk for just a minute about event handler response types. As we mentioned before, a response 
type determines exactly what the event handler is supposed to do when its event is generated. Several of 
the built-in response types are used for navigation to and from forms such as the following:  

•  Goto Form - Use this response type: 

o  When you want to close the current form and open a new form.  

•  Run Form - Use this response type: 

o  To open a new form while leaving the current form open. 
  With this type, both forms are available to the user.  

•  Run Form as Modal Child - Use this response type: 

o  When you want a new form opened, but in this case, the user cannot return to the original 

(parent) form until the child form is closed.  
  Typically, any changes made in the child form can then be pulled back into the parent 

form.  

•  Run Form as Linked Child - Use this response type: 

o  When you want both forms to stay open and be available to the user.  

  With this option, WinStudio maintains a link between the two forms, so that whatever 
change is made in one form is automatically reflected in the other form as well.  

•  Run Form to Retrieve Filter - Use this response type: 

o  To run the query form for the current form, while leaving the current form open.  

  When the user exits the query form, the filtered results can be returned to the original 

form.  

•  Goto Form Page - Use this response type: 

o  When you want the user to be able to move between form pages in a form that has them.  

  You will attach this type of event handler to the buttons you create on the HomePage, to 
provide for the navigation between the three form pages in the Expense Report portal 
form.

Multiple collections on forms  

On some of the forms we have used in previous lessons, you might have noticed that there can be more 
collections than one displayed on a form. Often, these collections are related in some way, so that the 
data displayed in one grid or field is somehow derived from the data displayed in another grid or field. But 
the collections also might not be related. This is possible because there are multiple collection types 
available for use on a form including: 

•  Primary collection 
•  Secondary collection 
•  Subcollection 

Primary collection 

The primary collection is the main top-level collection employed on a form. This is the collection that 
displays the data that is considered of primary importance in the form display. It is sometimes referred to 
as the “parent” collection.  

Secondary collection 

A secondary collection is another top-level collection used on a form, but one that is not directly linked to 
a primary collection.  

Subcollection 

A subcollection is a next-level collection that derives its data in some way from the primary or secondary 
collection. For example, if the primary collection displays invoice records, the subcollection might display 
the individual line items associated with a particular invoice.

Form security 

Until now the form has been created and tested using the Site Developer editing permissions. This means 
that the user account with Site Developer permissions has the new IDOs still checked out. Therefore, no 
other user can see the changes that have been made to the IDO, even though they are all created at this 
point. So the property classes and IDOs must be checked in before anyone else can see them. In order to 
test the portal form, you must check the IDOs and IDO property classes in so other users can use the 
new form. 

In order to test the portal form, you will set row-level filtering which requires you to use the Group form to 
set security on the MGCExpenseReports form. The system includes preexisting security groups but for 
the portal form you need to create a specific security group. When the new group is created using the 
Groups form, the goal is to restrict users to the portal page. 

As an administrator in the system, you can query any reports. As a portal user with no editing permissions 
and no super user rights, you can only query reports you create. 

Creating automatic row-level filtering 

The purpose of our Expense Report portal form is to provide a portal for employees to view, add, and 
modify expense reports. Because you want these users to be able to work only with their own expense 
reports, you need to find a way to automatically filter the expense reports entered into the system, so that 
only that user’s expense reports are presented. This not only allows users to work with their own expense 
reports, it also prevents them from accessing other users’ expense reports—and other users from 
accessing theirs.  

We have already built this into the portal for adding new reports. The Employee Number field is defaulted 
and hidden. But for the query reports facility, we also want to have a security-based policy to restrict 
normal users so they only see their reports. As it turns out, this is surprisingly easy to do using 
Mongoose’s built-in security features, specifically Groups and User Profiles. To accomplish this automatic 
filtering, you will use a combination of a Group profile you will call MGCPortalUsers that has limited 
access to issues for members of that group, specifically to the current user; and then you will assign a 
user to that group—MGCPortalTest.  

How to create the row-level filter 

You begin on the Groups form, to create a group that will act as an automatic filter to allow only the 
current user to view the expense reports created by the user.  

After you create the group, you will then use the Row Authorizations form to restrict which users can 
access certain records. In this case, you will be using the Row Authorizations form to restrict record 
access to the current user.  

Once the group with its row authorizations has been created, you assign users as members of that group.  

After the row-level filter is created you will also use the Group Authorizations form to further restrict the 
group’s access to just the forms in the Expense Reports portal; but you cannot do that yet, because the 
forms do not yet exist.

Mongoose multi-developer support and source 
control 

When developing your application, the Mongoose framework provides the ability for multiple developers 
to work on your application in the same environment without accidentally overwriting changes made by 
others. If you are extending an application from a vendor, Mongoose protects your application 
automatically during upgrades. 

Because a Mongoose application is mostly defined in metadata rather than code, Mongoose provides the 
typical check-in/check-out operations that developers utilize with code, but metadata is managed instead 
(i.e., locking, promoting, etc.). 

You can interface CloudSuite/Mongoose to a source control system. During check-in to a source control 
system Mongoose generates files representing the metadata and checks them into the system. 

Source control systems that Mongoose supports 

Although you will not work with a source control system during this course, Mongoose supports three 
popular source control systems:   

•  Microsoft® Visual SourceSafe®  
•  Microsoft Team Foundation Server  
•  Apache Subversion™  

Check-in/check-out 

Developers use the FormControl form or app server utility to check forms and global objects out before 
working on them. That procedure locks the checked out items to ensure that no one else will accidentally 
overwrite their work. When development is complete, the developer checks the items back in.  

Check-out procedure 

The check-out procedure copies the metadata from master to runtime, and marks them as locked in both 
places by the user. 

Check-in procedure 

The check-in procedure copies the metadata from runtime to master and unlocks them. If source control 
is configured, a SQL drop/create script is generated and checked into source control.

Checking IDOs and property classes in and out 

IDOs and property classes are also enabled for multi-developer check-in/check-out; however, the model 
is simpler. Within the single objects database, when an IDO is checked out, a separate row is created for 
it, marked as under development, and the production row is marked as locked (checked out). No one but 
the developer sees the changes being made until it is checked back in.  

The check-out operations for IDO metadata are done in the IDO editing forms. When check-in is done, if 
source control is enabled, Mongoose generates an IDO XML in the format we support for IDO 
import/export, and checks that into source control. 

For this course, you must check IDOs and property classes back in before the MGCPortalTest user will 
be able to view the forms that use them. You will not use an external source control system; however, 
Mongoose provides an internal mechanism to complete the check-in procedure for IDOs and property 
classes and you will do this in the next exercise.
