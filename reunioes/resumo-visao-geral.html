Reunião com Alberto Gómez Martínez · Participantes: João Quelhas, Daniel Santos
Resumo
🖥️ Interface do SyteLine ▶
•Grid (lista de registos) + Detalhe (dados do registo selecionado) + Botões de ação.
•Widgets: painéis com informação relacionada com o registo atual (ex: PDF do pedido no IDM).
•Inbox: notificações de workflows — aprovações, reatribuições, alertas.
•Os botões navegam entre forms (não abrem janelas separadas — ambiente de tabs/ecrãs).
👁️ O developer e o utilizador final veem exatamente o mesmo ecrã — qualquer alteração feita é imediatamente visível para o cliente.
🎨 Estrutura do Designer ▶
•Painel esquerdo: toolbar de componentes (botões, grids, campos, etc.).
•Área central: canvas do form — o que se vê é o que o utilizador vai ver.
•Painel direito: propriedades do componente selecionado (Design, Appearance, Behavior, Data, Source, Events).
•Para ver propriedades de uma Grid: clicar no botãozinho da grid — abre todas as tabs.
💡 Alterar um campo numa grid/form pode disparar um evento "Data Change". Ex: mudar "VendNam" → dispara "Data Change PO VenNam" → executa sequência de ações definida no Event Handler.
📝 FormScript — Linguagem e Estrutura ▶
•Linguagem: Visual Basic (VB) — todos os FormScripts no SyteLine são em VB.
•Global Scripts (bibliotecas partilhadas): podem ser em C#.
•Para navegar: usar o dropdown (lista todas as rotinas) ou Ctrl+F.
•Tipos de rotina:
Sub MinhaSubrotina() ' executa ações, não devolve valor End Sub Function MinhaFuncao() As Integer Return 0 ' 0 = sucesso, execução continua ' Return -1 = erro, execução para imediatamente End Function
⚙️ Eventos Standard (pré-definidos) ▶
•A maioria existe em pares (início + conclusão):
Standard Form CopyDisparado quando o utilizador clica em "Copiar".
Standard Form Copy CompleteDisparado quando a cópia termina.
Standard Form SaveQuando o utilizador clica em "Guardar".
Standard Object SaveQuando a coleção é de facto persistida na BD.
Standard Form PredisplayAntes de o form ser apresentado (sem par "Complete").
💡 Podes inserir lógica antes ou depois de qualquer evento standard usando a sequência de ações.
🚫 Como Impedir Apagar um Registo ▶
•Opção A — Proibir sempre: na coleção do form, em "Standard Operations", desmarcar "Delete". O botão fica sempre cinzento.
•Opção B — Validar condicionalmente: ao clicar em "Eliminar" o registo fica marcado com X vermelho (não apagado). Só é eliminado ao "Guardar".
⚙️ Na Opção B: no evento "Standard Object Save", verificar a condição — se não puder apagar, mostrar mensagem e devolver -1 para parar a execução.
⚡ Application Events (triggers da aplicação) ▶
•Funcionam como triggers ligados a tabelas/IDOs, independentes de um form específico.
IDO On Item InsertDurante o insert (antes de gravar).
IDO Post Item InsertApós o insert concluído.
IDO On Item DeleteDurante o delete.
IDO On Item UpdateDurante a atualização.
📌 Exemplo real: ao criar um cliente novo (Cursor Sequence == 0), enviar notificação ao grupo X + email automático.
📌 Exemplo real 2: evento "PDF by Email" — quando tarefa de impressão termina, envia email com o documento em anexo.
🕐 Background Tasks (tarefas agendadas) ▶
•Executam processos em segundo plano (normalmente à noite ou em intervalos definidos).
•Usos comuns: gerar/imprimir reports automaticamente, executar métodos de manutenção, correr Stored Procedures SQL.
•Tipo "Executable Type = Report": para reports. Tipo Method: para métodos/SPs.
🔄 Fluxo Típico de Desenvolvimento ▶
1Consultor (Luís Falcão / Landeiro) define o gap — o que há a fazer e em que form.
2Developer exporta backup via FormSync antes de tocar no form.
3Faz as alterações (designer + FormScript + eventos).
4Testa no ambiente de treino/demo.
5Avisa o consultor para testar do lado do cliente.
6Após validação, exporta o form e faz import em produção via FormSync.
